## Description
Define all shared TypeScript interfaces and types that represent the API response contracts consumed by the storefront landing page. These types are the contract between the backend API tickets (LP-003) and the frontend hook/component tickets (LP-004 onwards). They must be in `src/types/` so both the Express server and React client can import them.

## Acceptance Criteria
- [ ] `src/types/storefront.ts` created (named export, no default export)
- [ ] `HeroPromotion` interface exported: `{ headline: string; imageUrl: string; imageAlt: string; ctaLabel: string; ctaHref: string }`
- [ ] `StorefrontProduct` interface exported: `{ id: string; name: string; subtitle: string; imageUrl: string; price: number | null; stock: number; arrivedAt: string }`
  - `price: number | null` to handle the sold-out / missing-price state (UX A-02)
- [ ] `SportCategory` interface exported: `{ label: string; imageUrl: string; sportSlug: string }`
- [ ] `MembershipContent` interface exported: `{ heading: string; body: string; imageUrl: string }`
- [ ] `WishlistItem` type exported: `{ productId: string }`
- [ ] `CartItem` type exported: `{ productId: string; quantity: number }`
- [ ] All types have strict mode compliance — no `any`, no implicit any
- [ ] Types are co-located in `src/types/storefront.ts`; not scattered across component files

## Technical Notes
- Follow the existing `src/types/product.ts` pattern shown in `agents/sa/agent.md`
- Use `interface` for object shapes, `type` for aliases/unions
- `price: number | null` is intentional — the API returns `null` for sold-out items; the component renders "Sold Out" when `null`
- `CartItem` and `WishlistItem` may already exist if a prior feature created them; reuse and extend rather than duplicating

## Out of Scope
- Prisma schema types (those come from Prisma generation automatically)
- Full product types beyond what the landing page needs
- Any runtime validation (Zod schemas live in the API ticket LP-003)
