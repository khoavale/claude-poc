## Description
Implement the server-side wishlist endpoints (add / remove) and the corresponding React Query mutation hook on the frontend. Authentication is required — unauthenticated requests return HTTP 401. The wishlist Zustand store (LP-005) is the optimistic client state layer; this ticket adds the durable server-side persistence and the mutation that syncs them.

## Acceptance Criteria
- [ ] `POST /api/wishlist` accepts `{ productId: string }`, authenticates the caller, adds the product to the caller's wishlist, returns `{ productId, userId, createdAt }` with HTTP 201
- [ ] `DELETE /api/wishlist/:productId` authenticates the caller, removes the product from the wishlist, returns HTTP 204
- [ ] `GET /api/wishlist` authenticates the caller, returns the current wishlist item array for the user
- [ ] Unauthenticated requests to all three endpoints return HTTP 401 `{ message: 'Unauthenticated' }`
- [ ] Input validated with Zod: `productId` must be a non-empty string
- [ ] `src/api/wishlist.ts` created with: `addToWishlist(productId)`, `removeFromWishlist(productId)`, `fetchWishlist()`
- [ ] `src/api/hooks/useWishlist.ts` created with:
  - `useWishlist()` — `useQuery` for the current wishlist (enabled only when authenticated)
  - `useToggleWishlist()` — `useMutation` that calls add or remove based on current state, with optimistic update on `useWishlistStore` and React Query cache invalidation on settle
- [ ] Unauthenticated wishlist toggle in the UI (called from LP-007 product card) routes user to `/login` — the hook must return a flag `requiresAuth: boolean` so the component can handle this without knowing auth internals
- [ ] Unit tests cover: authenticated add, authenticated remove, unauthenticated returns 401

## Technical Notes
- Auth middleware: add an `authenticate` Express middleware (or reuse if one exists) — for now a stub that reads a session cookie or JWT header; the auth system itself is out of scope for this feature but the middleware contract must exist
- Wishlist table: create in LP-006-DB (migration ticket); this ticket depends on it
- Use `db.$transaction` if add-and-fetch is needed in a single operation; for add-only a single `db.wishlistItem.create` is fine
- `useToggleWishlist` must call `useWishlistStore.toggle` optimistically before the mutation resolves, then sync on success/error — this prevents UI flicker
